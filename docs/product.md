# Exocortex / 衍我 Product Note

## 一句话

衍我是一个本地优先的外部认知系统。它持续接收我的信息活动，形成可检索、可回放、可理解的个人上下文，让 AI 能和我共享同一份工作记忆。

## 产品立场

衍我不是一个聊天机器人，也不是一个普通知识库。

它更接近：

```text
我的外置认知层
```

它要做的事情是：

- 持续记录我在不同系统中的信息活动。
- 把这些活动整理成本地可信事实。
- 让 AI 在需要时能读取这些事实，理解我正在经历什么、做什么、推进什么。
- 最终辅助我回忆、决策、计划和执行。

## 当前阶段

现在只做衍我的一个很小部分：

```text
飞书消息同步
```

当前目标不是做完整衍我，也不是做最终 UI。当前目标是把一个核心能力做对：

```text
把我发过的消息、以及我收到的非免打扰会话消息，可靠同步到本地。
```

## 当前用户价值

第一阶段完成后，衍我至少能回答：

- 我最近在飞书说过什么？
- 我最近从非免打扰会话收到过什么？
- 某个时间段我的飞书消息流是什么样的？
- AI 能否基于这些消息理解我最近的工作上下文？

这不是“洞察”阶段，而是“记忆形成”阶段。

## 当前功能范围

### 做

- 同步我发出的飞书消息。
- 同步我收到的、来自非免打扰会话的飞书消息。
- 本地持久化这些消息。
- 保留原始 payload，便于未来重新解释。
- 建立可推进、可恢复的同步状态。

### 不做

- 不建模“已读”。
- 不把“有权限看到”误认为“我读过”。
- 不把内部同步参数作为产品主界面。
- 不追求第一版 UI 完整。
- 不直接进入摘要、项目识别、任务提取等语义层。

## 重要概念

### 发出过的消息

我本人发送的飞书消息。

这是当前阶段最可靠的消息类型。

### 收到过的消息

别人发送到我同步范围内的消息。

当前同步范围是：

```text
非免打扰会话
```

当前默认只同步非免打扰群聊。单聊可以后续纳入，但不改变核心抽象。

### 已读

当前阶段没有“已读”概念。

原因：

- 飞书客户端产品里有已读状态，但 `lark-cli` 当前没有证明能稳定暴露“我已读过哪些消息”的一等接口。
- 用非免打扰、可见、活跃等代理信号直接命名为已读，会污染产品语义。

如果未来获得可靠的一等信号，再单独设计。

## 长期方向

飞书消息只是第一个 Source。长期衍我会接入更多信息源：

- 飞书文档
- 日历
- 邮件
- 会议纪要
- 浏览器阅读记录
- 本地文件
- 任务系统

因此当前设计不能只服务飞书消息。它应该沉淀出一个可复用的信息同步内核。

## 交互原则

当前长期采用 terminal-first。

除非出现非常明确的特殊情况，衍我在相当长一段时间内不做 UI，或只做极少量 UI。我们优先把 terminal 体验做到极致：

- 命令清晰。
- 输出可读。
- 状态可信。
- 错误可诊断。
- 数据可检查。
- 操作可自动化。
- AI 能直接读取和解释本地结果。

Terminal 日常操作入口必须极简：

```text
npm run help
```

默认只展示核心高频命令。完整命令目录作为辅助能力保留：

```text
npm run help -- --all
```

日常运行和故障判断见 `docs/operations.md`。

任何新增脚本都必须进入完整目录；测试应防止脚本存在但未登记。进入默认入口的命令必须非常克制，只放用户经常需要的核心操作。

这个原则的原因是：当前阶段的核心不是展示，而是同步、存储、诊断和可恢复性。Terminal 能更直接地暴露系统真实状态，也更适合早期快速校正抽象。

如果未来要做 UI，需要先证明：

- UI 解决了 terminal 很难解决的问题。
- UI 不反向污染核心抽象。
- UI 不掩盖同步和数据质量问题。
- UI 与系统诊断能力保持隔离。

## 产品界面分层

衍我的界面应分为两类。

### 衍我主界面

面向人的认知体验。

它应该展示：

- 最近进入衍我的信息流。
- 我发出的信息。
- 我收到的信息。
- 后续可被 AI 理解和组织的上下文。

它不应该首先展示：

- 数据库路径
- batch size
- lookback
- polling controls
- 原始错误堆栈

### 系统状态界面

面向工程诊断和早期验证。

它可以展示：

- 同步是否健康。
- 每个信息源是否在运行。
- 每个同步范围推进到哪里。
- 最近一次同步成功或失败。
- 错误详情。
- 运行记录。

系统状态界面必须和衍我主界面隔离。它是辅助面板，不是产品主体验。

## 设计原则

1. 本地优先。
2. 不丢原始事实。
3. 不用模糊概念污染数据模型。
4. 内部机制不直接暴露成产品概念。
5. 同步正确性优先于 UI 完整性。
6. 先形成可信记忆，再做语义理解。
7. Terminal-first；没有明确必要时不做 UI。
8. Restart-safe：进程长时间停摆后重启，必须从持久化 checkpoint 继续，不漏消息，不靠内存状态恢复。

## 当前成功标准

第一阶段成功不是页面好看，也不是指标很多。

第一阶段成功是：

```text
衍我能持续、可靠、本地化地记录我的飞书消息流。
```

具体验收：

- 同步状态可恢复。
- 长时间停摆后重启能继续从上次持久化进度追赶。
- 消息不重复入库。
- 失败不会错误推进进度。
- 我发出的消息能被同步。
- 非免打扰会话中别人发给我的消息能被同步。
- 本地数据能被 AI 读取。
